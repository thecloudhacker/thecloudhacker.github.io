ESA Breach

Space Missions are hard, but it seems that security is harder. In a latest security announcement, the European Space Agency have confirmed they have started criminal investigations after Scattered LapsusS Hunters have claimed to siphon 500GB of data from their systems.

The agency isn't having a good time of it with security after a hacker known as "888" hoovered 200GB of data over Christmas and this latest breach includes some extremely sensitive information including "operational procedures, spacecraft and mission details, subsystems documentation, and proprietary contractor data from ESA partners including SpaceX, Airbus Group, and Thales Alenia Space".

The scientific community has long been known to prioritise data accessibility over security, but the latest breach continues a trend of bad reports of security issues at the agency. As any Intel operative will tell you, any space-related company will be under the microscope from groups to find ways into data, systems, orbital platforms or supply chains.

The breach has claimed to have been possible via an unpatched vulnerability. We'll have to await a full breakdown (if it arrives) to know more.

AWS CodeBuild Flaw

The security company, Wiz, discovered a critical flaw in AWS CodeBuild ( dubbed CodeBreach by the team ) that could have potentially led to the complete takeover of AWS's own GitHub repositories and put every AWS account in the world at risk. The team noted it to be potentially worse than the SolarWinds supply chain attack.

Discovered back in August 2025, the team contacted AWS who then fixed the security hole in September 2025 before the system could be compromised by groups or agencies.

The team details the severity of the issue sternly in their finding report: "The issue allowed a complete takeover of key AWS GitHub repositories - most notably the AWS JavaScript SDK, a core library that powers the AWS Console. By exploiting CodeBreach, attackers could have injected malicious code to launch a platform-wide compromise, potentially affecting not just the countless applications depending on the SDK, but the Console itself, threatening every AWS account."

In a tale as old as time, a couple of characters in a Regex filter allowed unauthenticated attackers to infiltrate the build environment and leak privileged credentials. Once again, the CI/CD pipeline is in focus as a security pain point. Whilst no action is necessary in this instance to resolve the primary issue, the team are advising a range of security measures to prevent untrusted pull requests triggering privileged builds and securing your GitHub connection with unique fine-grained personal access tokens for each CodeBuild project.

A HUGE thanks need to go to the Wiz team by the world's user populous and a stern question for all our AWS account managers of "why the hell am I hearing about this in global security news months after the event and not from you directly?".

Palo Alto DoS Vulnerability

Palo Alto have announced a CVSS 7.7 vulnerability ( tracked as CVE-2026-0227 ) in the Global Protect PAN-OS software that permits a denial of service attack due to an improper check for exceptional conditions.

The issue affects:

  • PAN-OS 12.1 < 12.1.3-h3, < 12.1.4
  • PAN-OS 11.2 < 11.2.4-h15, < 11.2.7-h8, < 11.2.10-h2
  • PAN-OS 11.1 < 11.1.4-h27, < 11.1.6-h23, < 11.1.10-h9, < 11.1.13
  • PAN-OS 10.2 < 10.2.7-h32, < 10.2.10-h30, < 10.2.13-h18, < 10.2.16-h6, < 10.2.18-h1
  • PAN-OS 10.1 < 10.1.14-h20
  • Prisma Access 11.2 < 11.2.7-h8
  • Prisma Access 10.2 < 10.2.10-h29

Palo Alto confirmed the issue is applicable only to PAN-OS NGFW or Prisma Access configurations with an enabled GlobalProtect gateway or portal. The exceptional condition allows an unauthenticated attacker to cause a DoS, with repeated attempts resulting in the firewall entering maintenance mode.

Whilst presently the attack does not appear to be actively used in the wild (as yet), it's important to ensure systems are patched as soon as possible.

More n8n Woes

Obviously in the crosshairs of nefarious groups, n8n continues to suffer further woes with a discovery of supply chain attacks on eight packages on npm registry. Endor Labs have uncovered the first in a new wave of supply chain attacks on n8n and its community node ecosystem.

Behaving like any other n8n integration once it is installed, it displays the usual configuration screens and saves the Google account OAuth tokens in encrypted format to the n8n credential store. When the workflow is executed, however, it runs code to decrypt the stored tokens using n8n's master key and then exfiltrates the credentials to a remote server.

The list of currently malicious packages is as follows:

  • n8n-nodes-hfgjf-irtuinvcm-lasdqewriit (4,241 downloads, author: kakashi-hatake)
  • n8n-nodes-ggdv-hdfvcnnje-uyrokvbkl (1,657 downloads, author: kakashi-hatake)
  • n8n-nodes-vbmkajdsa-uehfitvv-ueqjhhhksdlkkmz (1,493 downloads, author: kakashi-hatake)
  • n8n-nodes-performance-metrics (752 downloads, author: hezi109)
  • n8n-nodes-gasdhgfuy-rejerw-ytjsadx (8,385 downloads, author: zabuza-momochi)
  • n8n-nodes-danev (5,525 downloads, author: dan_even_segler)
  • n8n-nodes-rooyai-model (1,731 downloads, author: haggags)
  • n8n-nodes-zalo-vietts (4,241 downloads, authors: vietts_code and diendh)

Previous warnings on community node security risks have been issues by n8n and on self-hosted n8n instances it is strongly advised to disable community nodes by setting N8N_COMMUNITY_PACKAGES_ENABLED to false.

It is likely automation systems such as n8n along with CI/CD tooling will increasingly come under attack as workloads on them integrate with so many different platforms and hold high permission credentials. It is a reminder that security boundaries are not just around running platforms, but encircle the whole developer and operational environment.

New n8n CVE

A new year, a new vulnerability in software the world is using merrily. The latest vulnerability in n8n comes hot on the heels of the previous CVE in December which also allowed attackers to execute commands.

The new vulnerability ( CVE-2025-68668 ) is classified as a Protection Mechanism Failure, rated at 9.9 on the "oh crud!" meter. Affecting versions 1.0.0 through to 2.0.0, any authenticated user with permission to create or modify workflows to execute arbitrary operating system commands on the host running n8n

n8n have confirmed they implemented a "task runner-based native Python implementation in version 1.111.0 as an optional feaature for improved security isolation". The feature can be enabled using the N8N_RUNNERS_ENABLED and N8N_NATIVE_PYTHON_RUNNER environment variables prior to 2.0.0 where it is enabled by default.

As workarounds, n8n is recommending that users:

  1. Disable the Code Node by setting the environment variable NODES_EXCLUDE: "[\"n8n-nodes-base.code\"]"
  2. Disable Python support in the Code node by setting the environment variable N8N_PYTHON_ENABLED=false
  3. Configure n8n to use the task runner-based Python sandbox via the N8N_RUNNERS_ENABLED and N8N_NATIVE_PYTHON_RUNNER environment variables

As adoption of powerful and easy-to-use tooling becomes more normal, it's likely to find itself a target for higher scrutiny by nefarious parties.

Kimwolf Botnet

Brian Krebs has documented a new botnet discovered by Benjamin Brundage, the 22-year-old founder of the security firm Synthient, which has been dubbed Kimwolf. It has so far infected an estimated 2 Million devices globally, utilising flaws in residential proxy services.

The malware that seems to be targetting unofficial Android-based TV boxes and photo frames turns a user's Internet connection into a proxy node, allowing attackers to squirrel around on your LAN or use your Internet connection for botnet activities.

Synthinet has generated a page on their website to help users verify if their public IP has been tracked as a member of the Kimwolf botnet.

Brundage has also compiled a list of the unofficial Android boxes most highly represented in the botnet.

If you find or own a box matching one of the model names and/or numbers in the list, pull it out of your network and consider yeeting it into the fires of the scrap recycling centre.

MongoDB Bleeds Memory Data

A flaw in Zlib compression protocol header length ( CVE-2025-14847 ) which is utilised by default in MongoDB can return uninitialized heap memory without authenticating to the server.

Tracked by MongoDB and thankfully with a patch available ( MongoDB Jira Ticket), the MongoDB team urge users to patch immediately even though there are no current reports of active exploitation.

This data leak vulnerability affects the following versions:

  • MongoDB 8.2.0 through 8.2.3
  • MongoDB 8.0.0 through 8.0.16
  • MongoDB 7.0.0 through 7.0.26
  • MongoDB 6.0.0 through 6.0.26
  • MongoDB 5.0.0 through 5.0.31
  • MongoDB 4.4.0 through 4.4.29
  • All MongoDB Server v4.2 versions
  • All MongoDB Server v4.0 versions
  • All MongoDB Server v3.6 versions

The team have also advised "If you cannot upgrade immediately, disable zlib compression on the MongoDB Server by starting mongod or mongos with a networkMessageCompressors or a net.compression.compressors option that explicitly omits zlib. Example safe values include snappy,zstd or disabled"

Cisco AsyncOS Zero Day

'Twas the week before Christmas and all through the NOC, Cisco Admins were screaming and Criminals running amok.

Cisco have announced a critical vulnerability in the AsyncOS software under active exploitation by Chinese APT's. The attack campaign affects Cisco Secure Email Gateway systems (physical AND virtual) and Cisco Secure Web Manager appliances (physical AND virtual) of ALL release versions when these conditions are met:

  • The appliance is configured with the Spam Quarantine feature
  • The Spam Quarantine feature is exposed to and reachable from the Internet

Cisco have uncovered exploitation dating back to at least late November 2025 with the vulnerability being used to drop tunneling tools like ReverseSSH and Chisel. A lightweight Python backdoor has been also seen capable of receiving encoded commands and executing them. The CVE ( CVE-2025-20393 ) has a rating score of 10.0 and in the absence of a patch, administrators are being urged to restore their appliances to a secure configuration; limit access from the Internet; secure devices behind a firewall and only allow traffic from trusted hosts; separate mail and management functionality onto a separate network interface; monitor web log traffic for unexpected traffic and disable HTTP for the main administration portal.

New AWS Crypto Miner attack

It's been a thorn in the side for AWS-based companies for at least ten years now, and it's taken an interesting new turn. The deployment of crypto mining infrastructure from exposed or stolen keys has been a vast financial sink hole for countless companies, but Amazon have uncovered a new variant which has a sneaky attack mode targeting EC2 and ECS.

Once a key has been swiped by the criminal, resources and permissions are quickly enumerated before deploying the crypto mining resources across EC2 and ECS. The multi-stage attack begins with a discovery phase, running probes with the stolen IAM credentials to test permissions by using the --DryRun flag. This enables the attacker to validate IAM permissions without launching infrastructure, avoiding cost increase triggers and minimising forensic evidence.

The process continues with a CreateServiceLinkedRole and CreateRole phase to generate IAM roles for autoscaling groups and AWS Lambda, then attaching the AWSLambdaBasicExecutionRole policy. The ECS clusters deployed then register a malicious DockerHub image ( yenik65958/secret:user ) , creating a service using the task definition to initiate crypto mining on ECS Fargate nodes. The DockerHub image has been thankfully taken down ( after over 100,000 pulls ), however the data shows it configured to initiate a shell script on launch to start cryptocurrency mining using the RandomVIREL mining algorithm. The threat actor has also been observed to exploit EC2 service quotas to maximise resource consumption with scaling between 20 and 999 instances, targeting both high-performance GPU and machine learning instances as well as compute, memory and GP instances.

The attack also stands out for it's use of the ModifyInstanceAttribute action to enable termination protection, preventing instances from being terminated using the EC2 console, CLI or API. This action makes response to the incident more involved and shows an understanding of the standard clean-up practices.

Amazon reccomends GuardDuty is in place and enabled on all accounts and regions with Runtime monitoring enabled, along with integration to the Security Hub. As always, strong IAM controls are critically important with temporary keys instead of long-term access keys and enforced MFA on all accounts. Least privilege permissions should be the default as always and ship those logs to review easily as necessary by your teams!

Cloudflare Mitigates Largest Ever (so far) DDoS

Cloudflare have announced they detected and mitigated the largest ever (so far) DDoS measuring 29.7 terabits per second. Originating from a DDoS botnet-for-hire known as AISURU, the attack lasted for 69 seconds. Cloudflare did not disclose the target of this attack.

The AISURU botnet has prominently targeted telecoms providers, hosting providers, gaming companies and financial services. The attack was UDP bombarding an average of 15,000 destination ports per second, randomising various packet attributes in an attempt to evade detection and defences.

The growing DDoS problem has been seen hand-in-hand with the increase in malware, targeting and infecting as many devices as possible to utilise in these attacks. The usual kudos for having the largest botnet army is high on the agenda - more bots equals more likelyhood of causing issues for targets, equals higher price for hire.

Despite mitigating the incident in this battle, the war is yet to be over and we are likely to see an increased quantity of these high figure attacks in 2026.

NPM Woes Continue

I'd like to say this was just the Annus Horribilis for Node Package Manager, but I fear this is just the shape of things to come for the forseeable future. In today's ongoing saga of issues, North Korean threat actors behind the Contagious Interview campaign spammed the npm registry with 197 more malicious packages.

The team at Socket have been tracking the progress of the npm attacks which are designed to deliver a variant of OtterCookie. Once launched, the software attempts to evade any sandboxes and virtual machines, profiles the machine and establishes a command and control channel to provide the attackers with a remote shell, along with the ability to steal clipboard contents, log keystrokes, capture screenshots, gather browser credentials, documents, cryptocurrency wallet data and seed phrases.

The utilisation of these types of package attacks are likely to cause significant damage as they are being coupled with fake interview processes, a recent tactic to get developers to install and run code on their systems in the hope of completing technical assessment to secure a new role.

It's that time of year whereby people consider where they're heading in their career and starting to look elsewhere - just the ideal time for the threat actors to pretend to be a recruitment agency and have developers install malware. If that's undertaken on workplace machines, it might cause significant compromise or data theft at a time when teams are trying to wind-down for the holidays.

We'll have to see what further npm headlines appear in the first few months of 2026.

JSONformatter and CodeBeautify Data Leaks

You would have thought in the current climate of security consciousness that users would have enough savvy to not just blindly paste information into websites online that reformat and provide that information back on an open-access link, but you'd be wrong.

In the continuing case of "finding new ways to exfiltrate data by oblivious stupidity", years of corporate data has been found to have leaked (and remains) on the online tool sites JSONformatter and CodeBeautify. Cybersecurity company watchTowr Labs captured a dataset of over 80,000 files which contained: usernames and passwords; repository authentication keys; Active Directory, Database and FTP credentials; Cloud keys; LDAP configuration inforamtion; helpdesk API keys; meeting room API keys; SSH session recordings and all manner of personal information.

So, just a few items of interest, then.

One might consider that the data might be limited to developers running some early-doors testing, however the companies affected span critical national infrastructure, government, finance, insurance, banking, technology, retail, aerospace, telecommunications, healthcare, education, travel and cybersecurity sectors.

So, let's just say, the world.

In modern news there is so much emphasis placed on elite hackers breaking through tiny cracks in firewalls or security systems, whilst it appears a simple script scanning across online tooling can provide more readily availale data. This isn't so much a back door as a thousand acre farm with an "Open All Hours" neon sign at the open entrance.

Claude AI used for cyber espionage

In a security report from Anthropic the first catalogued usage of AI tools to attempt access to over thirty high-profile companies and government organisations. In some cases, the attacks was successful.

Traced-back to a Chinese state-sponsored group, the attackers used Claude Code and Model Context Protocol (MCP) to run the attacks without a human in the tactical execution loop. A human-developed framework utilised Claude to orchestrate multi-stage attacks, which were then in-turn carried out by several Claude sub-agents, all performing specific tasks. Once the sub-agents had developed exploit chains and their associated custom payloads, human operators spent between two and ten minutes reviewing the results of the AI's actions, then signing-off on the subsequent exploitations. The sub-agents ran to find and validate credentials, escalate privileges, moving laterally across the network, accessing and then stealing sensitive data. Post-exploitation, the human operator only had to again review the AI's work before approving the final data exfiltration.

"While we predicted these capabilities would continue to evolve, what has stood out to us is how quickly they have done so at scale" states Anthropic's new analysis. As with standard AI, it "requently overstated findings and occasionally fabricated data during autonomous operations". So, it's not just normal development teams suffering from the cocky and erroneous "I'm amazing!" responses from AI.

It may be the first documented case, but with the rise of automated systems it is unlikely to be the solitary example in the news.
